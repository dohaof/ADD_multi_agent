## 4. ADD Step 4 — Candidate Design Concepts, Comparison, and Selection

## 4.1 Candidate design concepts

### Alternative A — Layered monolithic backend plus relational database
A single deployable backend exposes REST APIs for all functions; an Angular frontend calls it directly; backend integrates with identity service and Channel Management System.

**Pros**
- Simpler initial delivery for **CON-4**
- Easier coordination for MVP
- Clear single deployment unit

**Cons**
- Weaker support for **QA-4 scalability** on query-heavy workloads
- Harder to isolate availability risks for **QA-3**
- Harder to support **QA-6 modifiability** for new protocols without touching core backend
- Less natural fit for **QA-2 reliability** and **QA-8 monitorability** around publication flow
- Less aligned with leveraging **Kafka** per **CRN-2**

---

### Alternative B — Service-based architecture with separate command/query/admin services and event-driven publication
An Angular frontend uses REST APIs. Backend is decomposed into services by responsibility. Price changes generate publication events carried via Kafka to decouple processing and support monitoring and reliability. Query capability is separated from command/admin logic.

**Pros**
- Better support for **QA-3 availability** and **QA-4 scalability** by isolating query path
- Better fit for **QA-2 reliability** using asynchronous publication and durable messaging
- Better support for **QA-6 modifiability** by separating core logic from protocol adapters
- Better team work allocation per **CRN-3**
- Leverages known technologies from **CRN-2**
- Aligns with **CON-6 cloud-native**
- Better support for **QA-8 monitorability** and **QA-9 testability**

**Cons**
- More operational complexity
- More interfaces to define
- Greater risk to MVP if decomposition becomes too fine-grained under **CON-4**

---

### Alternative C — Fully event-sourced/microservices architecture
All changes are persisted as events; multiple bounded services react asynchronously; read models are built from event streams.

**Pros**
- Strong audit/reliability structure
- Strong decoupling and scalability
- Natural fit for event streaming

**Cons**
- High complexity for first iteration
- Risky for **CON-4 MVP in 2 months**
- Could introduce technical debt through premature complexity, violating **CRN-4**
- Not necessary based only on provided requirements

## 2. Evaluation of the selected design concept against quality attributes

## 2.1 QA-1 Performance
**Scenario:** a base rate price is changed for a specific hotel and date during normal operation; prices for all rates and room types for the hotel are **published (ready for query)** in **<100 ms**.

### Positive aspects
The proposal helps performance by:
- separating **Pricing Command Service** from **Pricing Query Service**
- keeping a dedicated path for command processing
- making recalculation explicit in the command service
- isolating query access to a query-ready store/path

These are good structural choices.

### Risk / concern
There is a critical ambiguity:

- In responsibilities, the **Pricing Command Service** persists changed prices and emits an event.
- The **Price Publication Service** then “marks/persists published prices for query.”
- The scenario says prices must be **published (ready for query)** in under 100 ms.

This means the architecture currently implies that **query readiness depends on asynchronous downstream processing** by Kafka + Price Publication Service.

That creates uncertainty about whether the required <100 ms can be guaranteed.

Also, the proposal says:
- “When the prices are changed, they are pushed to the Channel Management System and become available for querying by external systems.”
- QA-2 requires both:
  - available for query successfully
  - also received by Channel Management System

The proposal does not clearly define whether:
- “ready for query” occurs before CMS acknowledgement,
- or only after CMS acknowledgement.

This matters directly for QA-1 and QA-2.

### Verification judgement
- **Partially addressed**
- The structure may support QA-1, but the proposal has **not yet shown enough** to conclude that the scenario is satisfied.

### Requested clarification
The architect should explicitly define the publication lifecycle states, especially:
- persisted
- ready for query
- sent to CMS
- acknowledged by CMS

A corrective conceptual view would look like this:

```mermaid
stateDiagram-v2
    [*] --> PriceChangeAccepted
    PriceChangeAccepted --> PricesCalculated
    PricesCalculated --> QueryReadyPublished
    QueryReadyPublished --> CMSPushAttempted
```

This clarification is necessary to make **QA-1, QA-2, and QA-8 analyzable**.

I also accept the Quality Agent’s concern that **authorization must be enforced in backend interfaces, not only in the UI**, and that **the query path must be structurally independent of CMS availability** after prices become query-ready.

## 2.2 Reconciliation decisions where Architect and Quality views differed

**Decision:** Adopt the Quality Agent’s clarification request.

- CMS receipt acknowledgement.

- **QA-1** requiring very fast “ready for query” behavior,
- **QA-3** query availability,
- **QA-4** query scalability,
while still preserving Kafka for **QA-2** reliability and **QA-8** monitorability of CMS publication.

**Reason:**  
This is necessary to satisfy **QA-5**. UI filtering alone is insufficient for secure access control.

**Decision:** Yes, this is a real risk; however, the decomposition is still accepted as the architectural structure for Iteration 1.

The structure is valuable for correctness, modifiability, and team allocation, but implementation/deployment granularity must be refined later to remain feasible for the **2-month MVP**.

## Step 4 — Choose Design Concepts, Compare Alternatives, and Select One

### Design Problem to Solve
We need a structure that simultaneously supports:
- broad business functionality,
- fast price publication,
- scalable/highly available queries,
- secure access control,
- future protocol extensibility,
- cloud-native deployment,
- team familiarity,
- independent integration testing.

### Candidate Design Concepts

#### Alternative A — Layered Monolith
A single deployable backend, internally layered into presentation/API, application, domain, and infrastructure.

**Strengths**
- Simpler to implement quickly for MVP (**CON-4**)
- Lower operational complexity
- Avoids unnecessary early distribution (**CRN-4**)

**Weaknesses**
- Harder to isolate independently scalable query load (**QA-4**)
- Harder to separate write/publication concerns from query availability (**QA-1, QA-3**)
- Adding alternative protocols without affecting core may be more difficult unless carefully structured (**QA-6**)
- Tighter coupling to external integrations may reduce testability (**QA-9**)

#### Alternative B — Microservice decomposition by capability
Separate deployable services for authentication facade, pricing changes, query, hotel management, rate management, user management, channel integration, etc.

**Strengths**
- Strong deployment and scaling isolation (**QA-3, QA-4**)
- Clear team allocation (**CRN-3**)
- Good protocol evolution possibilities (**QA-6**)

**Weaknesses**
- High operational and integration complexity for a greenfield replacement under short delivery timeline (**CON-4**)
- Increased distributed reliability complexity for guaranteed publication (**QA-2**)
- Risk of premature decomposition and technical debt through over-distribution (**CRN-4**)

#### Alternative C — Modular cloud-native backend with separated query/write responsibilities and event-driven integration
A browser-based Angular frontend with a Java-based backend organized as deployable services around major access patterns:
- one **command-oriented pricing/administration backend**
- one **query-oriented pricing API/backend**
- asynchronous eventing for publication/integration using Kafka
- external adapters isolated at the edges
- internal modular decomposition to preserve clear responsibilities

**Strengths**
- Supports independent scaling and availability of queries (**QA-3, QA-4**)
- Supports fast publication pipeline and asynchronous external delivery observation (**QA-1, QA-2, QA-8**)
- Isolates query protocol adapters from core query logic (**QA-6, CON-5**)
- Keeps decomposition moderate enough for 2-month MVP / 6-month release (**CON-4**)
- Leverages Java, Angular, Kafka knowledge (**CRN-2**)
- Cloud-native and testable through explicit external adapters (**CON-6, QA-9**)

**Weaknesses**
- More complex than a pure monolith
- Requires careful interface definition between write side, query side, and eventing

### Selected Design Concept
**Alternative C — Modular cloud-native backend with separated query/write responsibilities and event-driven integration**

### Rationale for Selection
This concept best satisfies the selected drivers:

- **QA-1 Performance**: price change handling and publication path are separated from general query load.
- **QA-2 Reliability**: event-driven integration with explicit publication responsibility supports reliable delivery tracking.
- **QA-3 Availability / QA-4 Scalability**: query-facing capability can scale independently from administrative and price-change workloads.
- **QA-5 Security**: centralized frontend/backend authorization integration with cloud identity service.
- **QA-6 Modifiability**: protocol adapters can be added around a stable query core.
- **QA-8 Monitorability**: publication events create observable points for complete measurement.
- **QA-9 Testability**: external systems are accessed through explicit interfaces/adapters.
- **CRN-2**: aligns with Java, Angular, Kafka.
- **CON-4**: avoids excessive microservice granularity.
- **CON-5**: REST now, other protocols later via adapter boundary.
- **CON-6**: compatible with cloud-native deployment.

1. **“100%” delivery semantics are asserted but not structurally supported in the proposal.**
   The design does not specify:
   - what happens if Channel Management System is unavailable,
   - how failed deliveries are detected,
   - how the system distinguishes “published internally” from “received by CMS,”
   - whether publication completion depends on CMS acknowledgment.

2. The sequence diagram shows a straightforward event flow, but **no reliability control loop** is visible.

3. There is no explicit responsibility allocated for:
   - tracking publication attempts,
   - tracking reception success/failure,
   - reconciling undelivered changes.

4. The monitoring component collects metrics, but **metrics alone do not guarantee reliability**.

### Assessment
**Structurally suggested but insufficiently supported. High risk.**

### Recommendation
The proposal should explicitly allocate a responsibility for **publication outcome tracking** and show the distinction between:
- internal publication ready for query,
- external push to CMS,
- confirmed receipt by CMS.

Minimal clarification view:

```mermaid
flowchart LR
    EV[Publication/Eventing] --> CMA[Channel Management Adapter]
    CMA --> CMS[Channel Management System]
    CMA --> PST[Publication Status Tracking]
    PST --> MON[Monitoring/Telemetry]
```

Without this, QA-2 cannot be judged as satisfied.

---

### QA-3 Availability
**Scenario:** Pricing query uptime SLA is 99.9% outside maintenance windows.

### Positive aspects
- Query responsibilities are separated from command/admin responsibilities.
- This separation supports fault isolation in principle.

### Gaps / risks
1. The proposal uses a shared generic **Persistence Layer** for all services.
   - This creates a potential **single point of failure** for query availability.
   - If hotel/rate/user/command workloads affect the same persistence element, query SLA risk remains.

2. No view identifies which elements are required to be available for **query-only operation**.
   - If the frontend, admin services, command service, Kafka, or CMS are down, can query still remain available?
   - The proposal implies yes, but does not make it explicit.

3. Maintenance windows are mentioned in the SLA, but the architecture does not distinguish between:
   - query plane,
   - admin/change plane,
   - integration plane.

### Assessment
**Structurally favorable but under-specified.**

### Recommendation
The proposal should explicitly record that the **query path must remain operational independently of price change processing and CMS integration failures**. This is particularly important because the selected concept’s value depends on that isolation.

Suggested clarifying availability dependency view:

```mermaid
flowchart TD
    PQS[Pricing Query Service] --> QDB[(Query-Ready Price Data)]
    External[External Query Client] --> PQS
    FE[Angular Frontend] --> PQS

    CMD[Pricing Command Service]
    KAFKA[Kafka]
    CMA[Channel Management Adapter]
    CMS[Channel Management System]

    CMD -. not required for query serving .-> PQS
    KAFKA -. not required for query serving .-> PQS
    CMA -. not required for query serving .-> PQS
    CMS -. not required for query serving .-> PQS
```

### QA-4 Scalability
**Scenario:** Scale from 100,000 to 1,000,000 daily price queries with no more than 20% average latency degradation.

### Positive aspects
- Dedicated **Pricing Query Service** is the strongest part of the proposal for this scenario.
- REST query adapter separation supports independent evolution.
- Query/read isolation from admin and command loads is directionally correct.

### Gaps / risks
1. Shared persistence again weakens the ability to verify scaling behavior.
2. The proposal does not define whether:
   - external and UI queries hit the same interface and same backing data path,
   - query protocol adapters can scale independently,
   - admin reads share the same operational path as heavy external queries.

3. No explicit capacity-sensitive boundary is shown between:
   - query protocol endpoint,
   - query application logic,
   - query persistence.

### Assessment
**Reasonably supported at a coarse level, but not yet strong enough to claim risk is under control.**

### Recommendation
Record explicit design intent that **query traffic is independently scalable from command/admin traffic** and identify which elements may scale for query load without requiring changes to core logic.

### QA-5 Security
**Scenario:** User logs in through the frontend; credentials validated against cloud identity service; once logged in, only authorized functions are presented.

### Positive aspects
- Use of cloud provider identity service directly satisfies a major constraint.
- User Administration Service allocates permission management responsibility.
- Frontend function visibility is identified.

### Gaps / risks
1. The proposal places too much emphasis on **frontend-based authorization presentation**.
   The scenario says users should be presented only with functions they are authorized to use, but the use cases also require:
   - users can only query/change **authorized hotels**.
   
   The proposal does not clearly state that **backend services enforce hotel/function authorization**, only that FE retrieves permissions and displays functions.

2. There is no explicit authorization boundary on:
   - Pricing Command Service,
   - Pricing Query Service,
   - Hotel/Rate/User administration services.

3. External query clients are shown contacting the Pricing Query Service directly, but the proposal does not clarify:
   - whether they are authenticated,
   - whether they are authorized,
   - whether same hotel-level restrictions apply.

### Assessment
**Partially addressed. Security enforcement is under-specified.**

### Recommendation
The proposal should explicitly record that authorization is enforced in backend service interfaces, not only in the UI. This is necessary to satisfy HPS-1 and QA-5.

Clarifying responsibility view:

```mermaid
    AUTHZ --> HAS[Hotel Admin Service]
```

This diagram is a correction of responsibility emphasis, not a redesign.

### Gaps / risks

### QA-8 Monitorability

### Gaps / risks

## 5. Key Risks and Tradeoffs

### Risk R1 — Query-ready publication path is missing
This is the most important gap because it affects:
- **QA-1 Performance**
- **QA-2 Reliability**
- **QA-3 Availability**
- **QA-8 Monitorability**

Without an explicit publication-to-query-readiness mechanism, the architecture cannot be validated against the most critical use case.

### Risk R2 — Security enforcement is biased toward UI visibility
This affects:
- **QA-5 Security**
- **HPS-1**
- **HPS-2**
- **HPS-3**
- **HPS-4/HPS-5/HPS-6**

Backend enforcement for hotel/function authorization is not explicit.

### Risk R3 — Shared persistence may undermine query SLA and scalability
This affects:
- **QA-3 Availability**
- **QA-4 Scalability**
- indirectly **QA-1 Performance**

The proposal claims separation of command and query responsibilities, but uses a generic persistence element that may collapse those benefits.

### Risk R4 — Reliability to Channel Management System is not demonstrated
This affects:
- **QA-2 Reliability**
- **QA-8 Monitorability**
- **QA-9 Testability**

The adapter alone is insufficient; outcome tracking responsibility is missing.

### Risk R5 — Deployability is uncovered
This affects:
- **QA-7 Deployability**
- **CRN-5**

Because it was omitted as a driver, the architecture has no recorded decision enabling no-code environment movement.

### Tradeoff T1 — Moderate decomposition vs delivery speed
The chosen concept balances scalability/availability against schedule, which is good. But the proposal should be careful not to let “service” naming imply unnecessary deployment fragmentation for the MVP.

### Tradeoff T2 — Event-driven decoupling vs reliability complexity
Kafka-based decoupling supports scaling and observability, but it introduces additional reliability obligations that the current design does not yet allocate.

These are **verification-oriented improvements**, not a redesign:

3. **Explicitly allocate responsibility for publication outcome tracking** to support QA-2 and QA-8.

6. **Clarify whether query availability is independent of command/CMS/eventing failures** to support QA-3.

### What is acceptable
- The selected overall concept is **reasonable**.
- Functional decomposition is broadly aligned with the primary use cases.
- Separation of query and command responsibilities is a good structural direction.
- Use of Angular, Java-oriented backend structure, and Kafka aligns with team knowledge and constraints.

### What is not yet sufficient
The proposal is **not yet fully verifiable** against the most critical quality scenarios because it lacks explicit support for:
- query-ready publication semantics,

- deployability requirements,

**Conditionally accept with required corrections/clarifications.**

## Step 4 — Choose Design Concepts That Satisfy the Selected Drivers

This step compares candidate design concepts for the selected elements.

---

### 4.1 Price Publication Reliability

#### Candidate A — Direct synchronous update from Price Change logic to Channel Management System
- **How it works:** Once prices are changed, the application directly calls the Channel Management System REST API as part of the request flow.
- **Advantages:**
  - Simpler end-to-end flow.
  - Fewer moving parts.
- **Disadvantages:**
  - Tight runtime coupling to external system availability.
  - If Channel Management System is unavailable, price change publication reliability is threatened.
  - Harder to achieve **QA-2**.
  - Harder to support independent testing for **QA-9** because the external dependency is in the critical path.
  - Weakens availability isolation between internal change processing and external publication.

#### Candidate B — Asynchronous event-driven publication using Kafka between internal pricing update and Channel Management adapter
- **How it works:** Price changes are committed internally, a publication event is produced, and a separate publication component consumes the event and pushes changes to Channel Management System via REST.
- **Advantages:**
  - Decouples internal price change processing from external system failures.
  - Better supports reliable delivery workflows and publication monitoring.
  - Aligns with **CRN-2** because the team knows Kafka.
  - Supports testability by isolating external integration behind an adapter.
  - Fits cloud-native style (**CON-6**).
- **Disadvantages:**
  - More components to operate.
  - More design complexity than direct synchronous integration.

#### Selection
**Selected: Candidate B — Asynchronous event-driven publication using Kafka**

#### Rationale
This best addresses:
- **QA-2 Reliability** by decoupling publication from direct external availability,
- **QA-8 Monitorability** by creating clear publication stages that can be measured,
- **QA-9 Testability** by isolating external dependencies,
- **CRN-2** by leveraging Kafka expertise,
- **CON-6** by fitting cloud-native design,
- while remaining compatible with initial REST interaction under **CON-5**.

---

### 4.2 Query Availability

#### Candidate A — Single combined application serving all functions
- **How it works:** Login, price changes, query, hotel/rate/user management, and publication coordination all run in one deployable backend.
- **Advantages:**
  - Simpler initial packaging.
  - Less integration overhead.
- **Disadvantages:**
  - Query availability is affected by failures in administrative or publication-related functions.
  - Poor isolation for **QA-3**.
  - Harder to allocate work cleanly (**CRN-3**).

#### Candidate B — Separate query-focused service from command/publication-oriented service
- **How it works:** Price query responsibilities are deployed in an element distinct from change/publish responsibilities.
- **Advantages:**
  - Better fault isolation for **QA-3**.
  - Better scaling and deployment independence.
  - Cleaner ownership boundaries for the team.
  - Supports future protocol extension for query endpoints without changing core pricing logic, helping later with **QA-6**.
- **Disadvantages:**
  - Requires explicit data/interface separation.
  - Slightly more architecture upfront.

#### Selection
**Selected: Candidate B — Separate query-focused service from command/publication-oriented service**

#### Rationale
This structure better supports:
- **QA-3 Availability** through fault isolation,
- **CRN-1** by establishing a robust initial structure,
- **CRN-3** by enabling work allocation,
- **CRN-4** by avoiding a tightly coupled monolith around critical paths,
- **CON-6** through cloud-native decomposition.

## QA-3 Availability — Reasonable structural support, but not yet demonstrated

### Strengths
The proposal supports availability through:

### Gap
The proposal does not yet show:

### Verification result
